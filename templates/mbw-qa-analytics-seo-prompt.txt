STRICT MODE STANDARD
This is a strict production QA workflow. Do not mark a page ready to go live unless the full checklist passes. If any item fails, clearly list the blocker and return corrected files whenever possible.

MOST IMPORTANT PRIORITIES
1. Functional EN/ES language switch
2. Centralized analytics through /assets/js/site.js
3. Strong SEO accuracy without unnecessary title or description rewrites

5. Global CSS usage wherever possible

7. Spanish accents only in user-facing Spanish copy and metadata

GLOBAL FILE ARCHITECTURE
Analytics execution must be centralized in:
/assets/js/site.js

Head asset helpers must remain in:
/assets/js/head.js

Shared styling should remain in:
/assets/css/site.css
/assets/css/header.css
/assets/css/footer.css

Page files should not duplicate analytics logic for:
- GA4 loading
- gtag initialization
- page_view_enhanced
- scroll_depth
- click listeners for [data-analytics-event]
- form tracking
- FAQ tracking

head.js responsibilities only:
- shared head assets
- cache-busted header.css
- font preconnects and font stylesheet
- favicon, Apple touch icon, manifest
- safe default metadata fallbacks only when missing

head.js must NOT:
- load Google Analytics
- initialize gtag
- add click listeners
- send analytics events

site.js responsibilities:
- load GA4 with measurement ID G-1ZYLW22XWP
- expose window.gtag
- send page_view_enhanced
- send scroll_depth at 25, 50, 75, and 90 percent
- listen globally for [data-analytics-event] clicks
- track language switch, CTAs, internal links, outbound links, WhatsApp links, phone/email links, forms, FAQ expands, cards, galleries, and page-type-specific events

LANGUAGE SWITCH STANDARD
Every bilingual page must have a raw EN/ES language switch immediately after #siteHeader and before hero/main content.

English page pattern:

<div id="siteHeader" data-current-lang="en"></div>
<nav class="rawLangLinks" aria-label="Language switch">
  <a href="FULL_EN_URL" lang="en" hreflang="en" aria-current="page" data-analytics-event="language_switch_click" data-analytics-label="English" data-analytics-location="language_switch" data-analytics-page-type="PAGE_TYPE" data-analytics-target-language="en">English</a>
  <a href="FULL_ES_URL" lang="es" hreflang="es" data-analytics-event="language_switch_click" data-analytics-label="Español" data-analytics-location="language_switch" data-analytics-page-type="PAGE_TYPE" data-analytics-target-language="es">Español</a>
</nav>

Spanish page pattern:

<div id="siteHeader" data-current-lang="es"></div>
<nav class="rawLangLinks" aria-label="Selector de idioma">
  <a href="FULL_EN_URL" lang="en" hreflang="en" data-analytics-event="language_switch_click" data-analytics-label="English" data-analytics-location="language_switch" data-analytics-page-type="PAGE_TYPE" data-analytics-target-language="en">English</a>
  <a href="FULL_ES_URL" lang="es" hreflang="es" aria-current="page" data-analytics-event="language_switch_click" data-analytics-label="Español" data-analytics-location="language_switch" data-analytics-page-type="PAGE_TYPE" data-analytics-target-language="es">Español</a>
</nav>

Language switch rules:
- Keep class exactly: rawLangLinks
- Use real HTML anchor tags
- Do not rely only on JavaScript-injected language links
- Use hreflang on both links
- Use aria-current="page" only on the active language
- Track both links with language_switch_click
- Keep the switch directly under #siteHeader
- Do not add page-level language switch CSS unless absolutely necessary
